# NeoSQLite v1.14.7 Release Notes

## TL;DR

NeoSQLite v1.14.7 is a **bug fix release** that resolves three critical aggregation pipeline failures causing application crashes when processing corrupted or malformed data, plus several SQL tier operator improvements.

**Key Fixes:**
- Aggregation pipelines now gracefully handle corrupted documents instead of crashing with `UnicodeDecodeError`, `OperationalError: no such column`, or `OperationalError: malformed JSON`.
- `sqlite3.ProgrammingError: Incorrect number of bindings supplied` errors are eliminated for `$project`, `$addFields`, `$setEquals`, and `$split` stages.
- `$rand` now runs in the SQL tier using SQLite's `RANDOM()` function.

- **Upgrade if you use:** Aggregation pipelines with `$facet`, `$sort`, `$lookup`, `$project`, or `$addFields` stages — especially on collections that may contain corrupted data or complex expressions.
- **Result Correctness:** Corrupted documents and unsupported operators are skipped gracefully with Python fallback — valid documents still produce correct results.

---

## Overview

This release focuses on aggregation pipeline resilience and SQL tier completeness:

- **$facet UTF-8 Decode Errors** — Fixed `UnicodeDecodeError` when `$facet` encounters non-UTF8 bytes in temporary table data column.
- **$sort Missing Column Errors** — Fixed `OperationalError: no such column: _id` when `$sort` processes intermediate tables without `_id` column.
- **$lookup Malformed JSON Errors** — Fixed `OperationalError: malformed JSON` when `$lookup` builds hash tables from collections with corrupted data.
- **SQL Binding Errors Eliminated** — Fixed `sqlite3.ProgrammingError: Incorrect number of bindings supplied` in `$project`, `$addFields`, `$setEquals`, and `$split`.
- **$rand SQL Tier Support** — `$rand` now runs natively in SQL tier using SQLite's `RANDOM()` function.
- **Tier Fallback Logging** — `NotImplementedError` for unsupported operators now logs at WARNING level (visible in API comparison runs) instead of ERROR with traceback.
- **Consistent Error Messages** — All unsupported operator messages standardized to "not supported in SQL tier".

## High Severity Fixes

### Fixed $facet Stage UTF-8 Decode Error

**Bug:** `$facet` stage crashed with `UnicodeDecodeError` when processing documents with non-UTF8 bytes in the data column.

**Root Cause:** When `$facet` extracts documents from temporary tables via `SELECT _id, data FROM {current_table}`, the `data` column may contain raw binary bytes instead of properly JSON-encoded text. The `neosqlite_json_loads()` function called `json.loads()` on these bytes, triggering the UTF-8 decode error.

**Solution:** Wrapped document loading in try/except — corrupted documents are skipped with a warning log, allowing the pipeline to continue processing valid documents.

**Impact:**
- **Before:** Single corrupted document → entire pipeline crashes → application error
- **After:** Corrupted documents skipped with warning → valid documents processed correctly

**Warning Log Example:**

```text
WARNING: Skipping corrupted document in $facet stage (id=1): 'utf-8' codec can't decode byte 0xdc in position 0
WARNING: $facet stage skipped 1 corrupted document(s) out of 10 total
```

---

### Fixed $sort Stage Missing Column Error

**Bug:** `$sort` stage crashed with `OperationalError: no such column: _id` when sorting intermediate tables that lack the `_id` column.

**Root Cause:** The `_process_sort_skip_limit_stage()` function builds a SELECT clause referencing `_id`, but previous stages (like `$facet` or certain transformations) may create temporary tables without that column. The column detection logic only handled the case where `_id` was missing but `id` existed — it didn't handle the case where both columns were absent.

**Solution:** Enhanced column detection with three-tier fallback:
1. If `_id` column missing but `id` exists: rewrite ORDER BY to use `id` via regex
2. If both columns missing: extract `_id` from JSON data using `json_extract(data, '$._id')`

**Impact:**
- **Before:** `$sort` after certain stages → `OperationalError` → pipeline failure
- **After:** `$sort` works regardless of intermediate table schema

---

### Fixed $lookup Stage Malformed JSON Error

**Bug:** `$lookup` stage crashed with `OperationalError: malformed JSON` when building hash tables from collections containing corrupted data.

**Root Cause:** The `_create_lookup_hash_table()` method uses SQL to extract join key fields via `json_extract(data, '$.field')`. If the `data` column contains malformed JSON (corrupted entries, improperly encoded Binary objects, or raw TEXT), SQLite's JSON functions fail on the entire query.

**Solution:** Three-layer defense:
1. **SQL Fallback:** When hash table creation fails with malformed JSON, falls back to row-by-row Python processing that skips corrupted documents
2. **Cursor JSONB Handling:** Updated `json_data_column()` to check `json_valid()` before calling `json()` on text values, preventing SQLite from failing on corrupted rows
3. **Document Loading:** Enhanced `_load_with_stored_id()` to catch JSON parse errors and return a minimal placeholder document

**Impact:**
- **Before:** Single malformed JSON in foreign collection → entire `$lookup` crashes
- **After:** Corrupted documents skipped → valid matches still found

---

### Fixed SQL Binding Errors in $project and $addFields

**Bug:** `$project` and `$addFields` stages crashed with `sqlite3.ProgrammingError: Incorrect number of bindings supplied` when expressions contained SQL parameters (`?`).

**Root Cause:** `json_data_column()` wraps expressions in `CASE WHEN typeof(expr)='blob' THEN json(expr) ...`, duplicating any `?` placeholders multiple times — causing a mismatch between placeholder count and parameter count.

**Solution:** When parameters exist, use a subquery to evaluate the expression once, then apply the `json_data_column()` wrapper in the outer query:

```sql
-- Before (6 placeholders, 1 param → crash)
SELECT id, CASE WHEN typeof(jsonb_object('key', ?))='blob' THEN json(jsonb_object('key', ?)) ... END AS data

-- After (1 placeholder, 1 param → works)
SELECT id, json(data) AS data FROM (SELECT id, jsonb_object('key', ?) AS data FROM table)
```

**Impact:** `$project` and `$addFields` with parameterized expressions (like `$let`, `$concat`, etc.) now work correctly without binding errors.

---

### Fixed SQL Binding Errors in $setEquals and $split

**Bug:** `$setEquals` and `$split` operators crashed with `sqlite3.ProgrammingError` due to placeholder duplication in their SQL implementations.

**Root Cause:**
- `$setEquals` uses NOT EXISTS clauses that reference operand SQL multiple times
- `$split` uses recursive CTEs that duplicate operand placeholders

**Solution:** Both operators now raise `NotImplementedError` to trigger Python fallback cleanly, matching the behavior of other operators that can't be reliably translated to SQL.

## Medium Severity Improvements

### $rand Now Runs in SQL Tier

**Enhancement:** `$rand` now uses SQLite's `RANDOM()` function instead of falling back to Python:

```python
# Returns random float in [0, 1)
{"$rand": {}}  →  ABS(RANDOM()) / 9223372036854775808.0
```

**Impact:** Faster random value generation in aggregation pipelines, no Python fallback overhead.

---

### Tier Fallback Logging at WARNING Level

**Enhancement:** When operators aren't yet supported in the SQL tier, the fallback to Python now logs at WARNING level instead of ERROR.

**Before:**

```text
ERROR: Temporary table aggregation error: Operator $regexMatch not supported in SQL tier
Traceback (most recent call last):
  File "..."
  NotImplementedError: Operator $regexMatch not supported in SQL tier
```

**After:**

```text
WARNING: SQL tier 2 aggregation fallback: Operator $regexMatch not supported in SQL tier
```

**Impact:** During API comparison runs (`./scripts/run-api-comparison.sh`), you can now see which operators are falling back to Python without the noisy traceback. Normal test runs remain quiet.

---

### Consistent Error Messages

**Enhancement:** All unsupported operator messages now use the format `"not supported in SQL tier"`:

```text
# Before: Inconsistent messages
"Operator $regexMatch requires Python fallback"
"$split requires Python fallback"
"Operator $concat not supported in SQL tier"

# After: Consistent format
"Operator $regexMatch not supported in SQL tier"
"Operator $split not supported in SQL tier"
"Operator $concat not supported in SQL tier"
```

---

### Unified Document Loading with Corruption Handling

**Problem:** UTF-8 decode errors occurred in multiple code paths — cursor loading, temporary table extraction, and hash table creation — each requiring separate fixes.

**Solution:** Extracted `_resolve_stored_id()` helper method to centralize `_id` resolution logic. Moved `bytes.decode("utf-8")` inside try/except in `_load_with_stored_id()`, returning a minimal `{"_id": ..., "__neosqlite_corrupted__": True}` placeholder for corrupted documents.

**Impact:** Eliminates code duplication, ensures consistent `_id` handling across all document loading paths, and makes future maintenance easier.

---

### _evaluate_sql_tier1 None Guard

**Fix:** When `_convert_expr_to_sql` returns `None` (for operators that fall back to Python), `_evaluate_sql_tier1` now checks for `None` before wrapping in `f"({sql_expr})"`, preventing the string `"(None)"` from being generated instead of proper `None` propagation.

## Migration Guide

### Upgrading from v1.14.6

This is a non-breaking bug fix release. All existing code continues to work without modification.

### For Users Experiencing Aggregation Crashes

If you were encountering these errors:

```text
UnicodeDecodeError: 'utf-8' codec can't decode byte 0xdc in position 0
sqlite3.OperationalError: no such column: _id
sqlite3.OperationalError: malformed JSON
sqlite3.ProgrammingError: Incorrect number of bindings supplied
```

**After v1.14.7:** These errors are resolved. Corrupted documents are skipped with warning logs, and valid documents are processed correctly:

```python
# Collection with some corrupted documents
collection.insert_many([
    {"name": "Alice", "score": 90},
    {"name": "Bob", "score": 85},
    # ... some documents may have corrupted data ...
])

# These pipelines now work without crashes
pipeline = [
    {"$match": {"score": {"$gte": 80}}},
    {"$facet": {
        "high_scores": [{"$sort": {"score": -1}}],
        "stats": [{"$count": "total"}]
    }}
]

results = list(collection.aggregate(pipeline))
# Valid documents processed, corrupted ones skipped with warnings
```

**Warning Monitoring:** During API comparison runs or development, you may see WARNING-level messages like:

```text
WARNING: Operator $regexMatch not supported in SQL tier
WARNING: SQL tier 2 aggregation fallback: Operator $split not supported in SQL tier
```

These indicate operators falling back to Python — results are still correct, just slightly slower.
