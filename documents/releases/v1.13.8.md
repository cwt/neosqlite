# NeoSQLite v1.13.8 Release Notes

## TL;DR

NeoSQLite v1.13.8 is a **security and reliability release** that fixes SQL injection vulnerabilities, adds context manager support for cursors, and improves transaction cleanup reliability.

- **Upgrade if you use:** changestreams, aggregation cursors, bulk operations, or accept user-provided collection/identifier names
- **No need to upgrade if:** Your deployment is internal with trusted inputs and your current version is stable
- **Breaking changes:** None

---

## Breaking Changes

**None** — v1.13.8 is fully backward compatible with v1.13.7.

---

## Overview

This release addresses critical security and reliability issues:

- **SQL injection prevention**: Collection names in changestream triggers are now validated, and comment markers in queries are escaped instead of removed
- **Cursor resource management**: Added context manager protocol (`with` statement) for guaranteed cleanup
- **Transaction cleanup reliability**: Added `finally` blocks to ensure savepoints are always released
- **Identifier validation**: Switched from broken bracket quoting to strict whitelist validation
- **Error logging**: Replaced silent failures with proper logging in migration cleanup

---

## Security Fixes

### Fixed: SQL Injection in Changestream Trigger Creation

**Problem**: Collection names were directly interpolated into `CREATE TRIGGER` and `DROP TRIGGER` statements, allowing SQL injection attacks via malicious collection names.

**Attack vector**:

```python
# Malicious collection name could execute arbitrary SQL
collection = db["users'; DROP TABLE users; --"]
# This would execute the DROP TABLE command!
```

**After (validated)**:

```python
# Collection names must match ^[A-Za-z_][A-Za-z0-9_]*$
collection = db["users'; DROP TABLE users; --"]  # Raises ValueError!
collection = db["valid_collection_name"]  # Works correctly
```

### Fixed: SQL Comment Marker Injection

**Problem**: Comment markers (`--` and `/**/`) were being removed from queries, allowing SQL injection through crafted input.

**Before (vulnerable)**:

```python
# User input: "normal -- DROP TABLE users"
# Result: "normalDROP TABLE users"  # Marker removed, attack preserved!
```

**After (escaped)**:

```python
# User input: "normal -- DROP TABLE users"
# Result: "normal - - DROP TABLE users"  # Marker escaped, attack neutralized!
```

**Examples**:

```python
# Attack attempts are now safely escaped
"normal -- DROP TABLE"  →  "normal - - DROP TABLE"
"/* malicious */"       →  "/ * malicious * /"
```

### Fixed: SQL Injection via Malicious Identifiers

**Problem**: The `quote_identifier()` function used SQLite bracket quoting (`[name]`) with `]]` escaping, but SQLite's `[[]]` escaping doesn't work as documented—`[[]]` raises "unrecognized token: ]" error.

**Before (broken)**:

```python
# This would fail or allow injection
quote_identifier("name]; DROP TABLE users; --")
```

**After (whitelist validation)**:

```python
# Only safe identifiers allowed: ^[A-Za-z_][A-Za-z0-9_]*$
quote_identifier("valid_name")      # Returns: [valid_name]
quote_identifier("123invalid")      # Raises ValueError
quote_identifier("name; DROP TABLE")  # Raises ValueError
```

## Bug Fixes

### Fixed: Changestream Race Conditions

**Problem**: Events could be lost or reprocessed if external cleanup deleted rows between poll and processing, or if the application crashed after fetching but before updating `_last_id`.

**Before (race condition)**:

```python
# Poll events
events = db.execute("SELECT * FROM changestream WHERE id > ?", (last_id,))
# <-- Application crash here loses events
# <-- External cleanup here causes reprocessing
db.execute("DELETE FROM changestream WHERE id <= ?", (last_id,))
```

**After (atomic read-delete)**:

```python
# BEGIN IMMEDIATE ensures atomicity
BEGIN IMMEDIATE;
SELECT * FROM changestream WHERE id > ?;
DELETE FROM changestream WHERE id <= ?;
COMMIT;  # Only update _last_id after successful commit
```

**Result**: Exactly-once processing semantics within a single changestream instance.

### Fixed: Cursor Resource Leaks

**Problem**: Cursor cleanup relied solely on `__del__`, which is unreliable in Python due to circular references, interpreter shutdown limitations, and no guarantee of execution.

**Before (unreliable cleanup)**:

```python
cursor = collection.find(query)
# If exception occurs or cursor is abandoned,
# temporary tables may never be dropped
```

**After (context manager)**:

```python
with collection.find(query) as cursor:
    for doc in cursor:
        process(doc)
# cursor.close() guaranteed to be called, even on exception
# Temporary tables from aggregation are properly dropped
```

**Usage**:

```python
# Explicit resource management
with collection.find({"status": "active"}) as cursor:
    for doc in cursor:
        handle(doc)
# Cleanup guaranteed
```

### Fixed: Savepoint Leaks in Bulk Operations

**Problem**: If `RELEASE SAVEPOINT` failed silently after a partial failure, the savepoint could remain in the transaction stack indefinitely, causing transaction state corruption.

**Before (potential leak)**:

```python
def _execute_ordered(operations):
    savepoint_name = f"sp_{id()}"
    db.execute(f"SAVEPOINT {savepoint_name}")
    try:
        # Execute operations...
        # If operation fails but RELEASE also fails, savepoint stays!
    except Exception:
        pass  # Silent failure
    # Savepoint never released!
```

**After (guaranteed cleanup)**:

```python
def _execute_ordered(operations):
    savepoint_name = f"sp_{id()}"
    released = False
    db.execute(f"SAVEPOINT {savepoint_name}")
    try:
        # Execute operations...
        db.execute(f"RELEASE {savepoint_name}")
        released = True
    except Exception:
        raise
    finally:
        if not released:
            # Always attempt release, preventing resource leaks
            db.execute(f"RELEASE {savepoint_name}")
```

### Fixed: Silent Migration Failures

**Problem**: Bare `except Exception: pass` blocks in `_migrate_to_autovacuum()` masked critical database errors, potentially leading to data corruption or inconsistent state.

**Before (silent failures)**:

```python
try:
    db.execute("PRAGMA wal_checkpoint(TRUNCATE)")
    db.commit()
    db.close()
except Exception:
    pass  # Silent failure - migration state unknown!
```

**After (logged failures)**:

```python
try:
    db.execute("PRAGMA wal_checkpoint(TRUNCATE)")
    db.commit()
    db.close()
except sqlite3.OperationalError as e:
    logger.warning(f"WAL checkpoint failed (expected for no WAL): {e}")
except Exception as e:
    logger.error(f"Migration cleanup failed: {e}", exc_info=True)
    raise
```

**Now logs**:
- `sqlite3.OperationalError` at WARNING level (expected for no WAL/no transaction)
- Other exceptions at ERROR level with full traceback

### Fixed: Linting Errors

- Updated lint script to match current standards
- Fixed all reported linting errors across the codebase

## New Features

### Added: Cursor Context Manager Protocol

**New**: Cursor class now supports `with` statement for guaranteed cleanup.

```python
# Before (relied on __del__)
cursor = collection.find(query)
try:
    for doc in cursor:
        process(doc)
finally:
    cursor.close()  # Might not be called in edge cases

# After (context manager)
with collection.find(query) as cursor:
    for doc in cursor:
        process(doc)
# cursor.close() always called, even on exception
```

**Benefits**:
- Guaranteed cleanup of temporary tables from aggregation
- Proper resource release even on exceptions
- Cleaner, more Pythonic code

## Migration Guide

### Upgrading from v1.13.7

**No breaking changes** — v1.13.8 is fully backward compatible.

### If You Use Changestreams

Your changestreams now have exactly-once processing semantics:

```python
# Existing code continues to work
cursor = collection.watch()
for change in cursor:
    handle(change)
```

### If You Use Aggregation Cursors

You can now use context managers for guaranteed cleanup:

```python
# New recommended pattern
with collection.aggregate(pipeline) as cursor:
    for doc in cursor:
        process(doc)
# Cleanup guaranteed
```

### If You Accept User Input for Collection Names

Collection names are now validated:

```python
# These now raise ValueError before SQL execution
db["users'; DROP TABLE users; --"]  # ValueError!
db["collection name with spaces"]   # ValueError!

# Use only safe names
db["valid_collection_name"]  # Works
```

## Known Limitations

### NX-27017 Limitations

- **No replication/sharding**: Coming never™ — This is NX-class, not NCC-1701!
- **Change streams via replica set**: Not supported (SQLite-trigger-based `watch()` is available!)
- **`find_raw_batches` with batch_size**: Requires cursor state management
