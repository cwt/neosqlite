## Major Feature: SQL Translation Caching

### The Problem

Every aggregation pipeline or `$expr` query must be translated from MongoDB-style syntax to SQL. This translation involves:
1. Parsing pipeline stages or expressions
2. Building CTE (Common Table Expression) chains or temp tables
3. Converting field paths
4. Generating SQL string

For applications that repeatedly run the same query structure (e.g., dashboard queries, periodic reports), this translation overhead accumulates.

### The Solution

Cache the translated **SQL template** using the query structure as the key. On cache hit, extract parameter values from the new query and execute immediately.

**Parameterized SQL Caching**:

```python
# Naive approach: Each size = different cache key
{"$sample": {"size": 3}}   → LIMIT 3   → Cache miss
{"$sample": {"size": 20}}  → LIMIT 20  → Cache miss

# Parameterized approach: Same template, different values
{"$sample": {"size": 3}}   → SQL: "LIMIT ?" + params: [3]
{"$sample": {"size": 20}}  → SQL: "LIMIT ?" + params: [20]
# Both use same cache entry!
```

### Supported Parameterized Operators

| Operator | Parameter | Example |
|----------|-----------|---------|
| `$sample` | `size` | `{"$sample": {"size": 10}}` |
| `$limit` | `limit` | `{"$limit": 100}` |
| `$skip` | `skip` | `{"$skip": 50}` |

### Configuration

```python
import neosqlite

# Default: cache enabled with 100 entries
conn = neosqlite.Connection('mydb.db')

# Custom cache size
conn = neosqlite.Connection('mydb.db', translation_cache=50)

# Disable cache (useful for development/debugging)
conn = neosqlite.Connection('mydb.db', translation_cache=0)
```

### Debug API

```python
users = conn.users

# Pipeline cache (Tier-1)
qe = users.query_engine.sql_tier_aggregator
qe.is_cache_enabled()          # True/False
qe.cache_size()                # Current entries
qe.get_cache_stats()           # Hit rate, hits, misses, etc.
qe.dump_cache()                # Debug cache contents
qe.clear_cache()               # Clear all entries

# $expr cache (Tier-2)

tier2.get_cache_stats()

```

## Tier Change Tracking Callbacks

### Why Use Tier Callbacks?

When the translation cache has a bug or queries unexpectedly fall back to slower tiers, the tier callback system helps you detect this:

- **Normal operation**: Queries should consistently use `tier1` or `tier1_standard` (both are SQL)
- **Problem detected**: If you see `tier1` → `tier2` → `tier3` changes unexpectedly, something is wrong

### Usage

```python
import neosqlite

tier_changes = []

def tier_callback(prev_tier, new_tier, pipeline):
    tier_changes.append((prev_tier, new_tier))
    if prev_tier and prev_tier != new_tier:
        print(f"WARNING: Tier fallback detected!")
        print(f"  {prev_tier} -> {new_tier}")
        print(f"  Pipeline: {pipeline}")

conn = neosqlite.Connection(':memory:', translation_cache=100)
users = conn.users
users.insert_many([{'age': i} for i in range(100)])

qe = users.query_engine
qe.add_tier_change_callback(tier_callback)

# Run queries
list(users.aggregate([{'$match': {'age': {'$gt': 50}}}]))
list(users.aggregate([{'$match': {'age': {'$gt': 25}}}]))  # Should hit cache

# Check if any fallback happened
print(f"Tier changes: {tier_changes}")
# Expected: [(None, 'tier1')] - no fallback!
```

### API

- `add_tier_change_callback(callback)`: Register a callback to be notified of tier changes
- `remove_tier_change_callback(callback)`: Remove a registered callback
- `get_last_tier()`: Get the last tier that was used
- `clear_tier_callbacks()`: Remove all registered callbacks

### Callback Signature

The callback receives three arguments:
- `previous_tier`: The tier used in the previous query (or `None` for first query)
- `new_tier`: The tier used for the current query
- `pipeline`: The aggregation pipeline that was executed

### Tier Types

| Tier | Description |
|------|-------------|
| `tier1` | SQL CTE-based aggregation (new optimizer) |
| `tier1_standard` | SQL non-CTE aggregation (older SQL builder) |
| `tier2` | Temporary table for complex queries |
| `tier3` | Python fallback for unsupported operations |

Note: `tier1_standard` was previously called `tier1_legacy`. Both tier1 and tier1_standard use single SQL execution, categorized as "SQL" in the 3-tier model.

## Temp Table Cleanup for Tier-2 Queries

Tier-2 `$expr` queries use temporary tables for evaluation. v1.11.0 adds proper cleanup:

- Cursor tracks `tables_to_cleanup` list
- `close()` and `__del__()` methods ensure cleanup on cursor exhaustion
- Cleanup chain: Connection → Collection → QueryEngine → QueryHelper → TempTableExprEvaluator

```python
# Automatic cleanup when cursor is exhausted or closed
cursor = users.find({"$expr": {"$gt": ["$age", 30]}})
for doc in cursor:
    print(doc)
# Temp tables automatically dropped when cursor closes
```

## MongoDB-Compatible Aggregate Command

v1.11.0 adds MongoDB-compatible `db.command("aggregate")` with explain support:

```python
# MongoDB-style aggregate command
result = conn.command(
    "aggregate",
    "collection_name",
    pipeline=[{"$match": {"status": "active"}}],
    explain=True  # Returns execution plan
)

# Returns
{
    "tier": 1,
    "type": "SQL Tier 1.5 (Non-CTE-based)",
    "sql": "SELECT ...",
    "params": [...],
    "sqlite_plan": [...]
}
```

**Removed**: `AggregationCursor.explain()` method — now uses `conn.command("aggregate", ..., explain=True)` for 100% MongoDB API compatibility.

## Bug Fixes

**Important Note on Correctness**: NeoSQLite's 3-tier architecture ensures 100% result accuracy. When SQL translation encounters issues, queries automatically fall back to Python (Tier-3), which provides complete MongoDB-compatible evaluation. The bugs below affected **performance** (causing slower execution via Python fallback), not **correctness** — v1.10.1 users always received accurate results.

### Fixed: Nested Field Path Extraction

Fixed `_extract_field_paths_from_dict()` to properly extract nested field paths like `$profile` and `$profile.age` from `{"profile": {"age": {"$gt": 25}}}`.

This affects SQL-tier query optimization for nested field queries in both `find()` and `aggregate()` operations. Previously, complex nested field queries might fall back to Python evaluation (Tier-3) instead of using optimized SQL (Tier-1), resulting in slower performance but still correct results.

### Fixed: $bucket and $count Parameter Extraction

- Fixed `$bucket` parameter extraction to properly handle boundary parameters
- Fixed `$count` parameter extraction (was previously ignored)
- Fixed `$facet` handling to return `None` instead of causing type error

These fixes improve SQL translation accuracy for aggregation pipelines using bucketing and counting operations. Previously, pipelines using these operators might fall back to Python evaluation instead of SQL optimization, but results remained 100% accurate.

## Known Limitations

### Translation Cache

**Query-Centric Design**: Cache keys are derived from query structure, not database schema. This is correct for schemaless document databases:

- When users change document fields (e.g., `status` → `state`), they must update queries
- Updated queries produce different cache keys → cache miss → fresh SQL generation
- No stale data risk since query structure drives cache invalidation

**Memory Usage**: Each entry stores a full SQL template. For most applications, 100 entries (default max size) uses negligible memory.

### SQL-Tier Limitations

**Fallback Required When**:
- Complex expressions can't be translated to SQL
- `$facet` stage (not supported in SQL tier)
- Deeply nested array transformations

### Performance Considerations

- Cache benefits repeated query patterns — unique queries each time won't benefit
- First query always incurs translation cost (cache miss)
- Subsequent identical-structure queries benefit (cache hit)
- Use API comparison benchmarks to identify optimization opportunities
