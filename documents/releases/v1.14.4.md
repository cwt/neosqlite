# NeoSQLite v1.14.4 Release Notes

## TL;DR

NeoSQLite v1.14.4 is a **comprehensive bug fix release** that resolves 5 critical aggregation pipeline issues including `$lookup` returning incorrect types, ObjectId parameter binding failures, type mismatches in join operations, and unsupported operators after `$group` stages.

**Critical Fixes:** This release eliminates silent data corruption from `$lookup` returning string `"[]"` instead of empty lists, fixes `sqlite3.ProgrammingError` when using ObjectId objects in `$match` parameters, and enables `$ne` operator support after `$group` stages.

- **Upgrade if you use:** `$lookup` aggregations, ObjectId fields in queries, or `$match` stages after `$group`.
- **Result Correctness:** Achieved **100% compatibility** in API comparison tests with comprehensive test coverage for all fixed scenarios.

## High Severity Fixes

### Fixed $lookup Returning String Instead of Empty List

**Problem:** When `$lookup` found no matching documents, the `as` field was returned as the string `"[]"` instead of an empty list `[]`. This caused silent type errors in downstream code that expected list operations.

**Root Cause:** The SQL `COALESCE` expression was inserting a literal string `'[]'` rather than a proper JSON array value.

**Solution:** Changed all `COALESCE(..., '[]')` expressions to `COALESCE(..., json('[]'))` in:
- Hash join implementation (`_process_lookup_hash_join`)
- Correlated subquery implementation (`_process_lookup_correlated_subquery`)
- Pipeline result table generation

**Impact:**
- **Before:** `comments = results[0].get('comments')` â†’ `type: str`, value: `"[]"`
- **After:** `comments = results[0].get('comments')` â†’ `type: list`, value: `[]`

**Example:**

```python
# This now returns correct type!
pipeline = [
    {'$match': {'_id': 1}},
    {'$lookup': {
        'from': 'comments',
        'localField': '_id',
        'foreignField': 'post_id',
        'as': 'comments'
    }}
]

results = list(db.posts.aggregate(pipeline))
comments = results[0].get('comments')

# Before v1.14.4: type(comments) == str, comments == "[]"
# After v1.14.4:  type(comments) == list, comments == []

# No more type checking needed!
for comment in comments:  # Works correctly now
    print(comment)
```

### Fixed ObjectId Type Cannot Be Bound as Parameter

**Problem:** Passing a `neosqlite.objectid.ObjectId` object directly in a `$match` stage caused `sqlite3.ProgrammingError: Error binding parameter N: type 'ObjectId' is not supported`.

**Root Cause:** SQLite's parameter binding doesn't understand custom Python ObjectId types, requiring string conversion.

**Solution:** Implemented `_sanitize_params()` helper function that automatically converts ObjectId instances to strings before SQL execution:

1. **Parameter Sanitization** â€” Added `_sanitize_params()` in `temporary_table_aggregation.py` to convert ObjectId â†’ str
2. **Automatic Application** â€” Applied sanitization in `_process_match_stage()` before parameter binding
3. **Transparent to Users** â€” No code changes required, works automatically

**Impact:**
- **Before:** `{'$match': {'_id': oid}}` â†’ `sqlite3.ProgrammingError`
- **After:** `{'$match': {'_id': oid}}` â†’ Works correctly

**Example:**

```python
from neosqlite.objectid import ObjectId

# This now works without manual conversion!
oid = ObjectId('669abc123def456789012345')
pipeline = [
    {'$match': {'_id': oid}},
    {'$lookup': {
        'from': 'comments',
        'localField': '_id',
        'foreignField': 'post_id',
        'as': 'comments'
    }}
]

results = list(db.posts.aggregate(pipeline))
# Returns matching documents with joined comments
```

**Workaround No Longer Needed:**

```python
# NO LONGER NEEDED - ObjectId binding now works natively

# Old workaround (v1.14.3 and earlier):
pipeline = [{'$match': {'_id': str(oid)}}]

# New approach (v1.14.4+):
pipeline = [{'$match': {'_id': oid}}]  # Just pass ObjectId directly
```

## Medium Severity Fixes

### Fixed $lookup Type Mismatch Returning Empty Results

**Problem:** When `$lookup` join fields had different types (ObjectId vs string), the join returned empty arrays silently. No error was raised, but results were incorrect.

**Root Cause:** ObjectId is stored as JSON object `{"__neosqlite_objectid__":true,"id":"<oid_string>"}` in the database. Direct string comparison failed because the full JSON object was being compared instead of extracting the actual ID string.

**Solution:** Implemented ObjectId-aware field extraction with `_json_extract_field_with_objectid_support()`:

1. **SQL Expression Generator** â€” Created helper function that detects ObjectId JSON structure and extracts the actual ID string
2. **Hash Join Updates** â€” Updated hash table creation to use ObjectId-aware extraction
3. **Correlated Subquery Updates** â€” Updated all join comparisons to use proper extraction
4. **Pipeline Support** â€” Applied to pipeline result table generation

**SQL Logic:**

```sql
-- Before: Compared full JSON object
CAST(json_extract(data, '$.post_id') AS TEXT)

-- After: Extracts actual ObjectId string
CASE 
  WHEN json_extract(json_extract(data, '$.post_id'), '$.__neosqlite_objectid__') = 1 THEN 
    json_extract(json_extract(data, '$.post_id'), '$.id')
  ELSE CAST(json_extract(data, '$.post_id') AS TEXT)
END
```

**Impact:**
- **Before:** ObjectId field joining with string field â†’ empty `[]`
- **After:** ObjectId field joining with string field â†’ correct matches via type coercion

**Example:**

```python
from neosqlite.objectid import ObjectId

post_oid = ObjectId('669abc123def456789012345')
db.posts.insert_one({'_id': post_oid, 'title': 'Test'})

# Comments with ObjectId reference (matching type)
db.comments.insert_one({'_id': 1, 'post_id': post_oid})

# Now correctly matches!
pipeline = [
    {'$match': {'_id': post_oid}},
    {'$lookup': {
        'from': 'comments',
        'localField': '_id',
        'foreignField': 'post_id',
        'as': 'comments'
    }}
]

results = list(db.posts.aggregate(pipeline))
assert len(results[0]['comments']) == 1  # Now works!
```

### Fixed $ne Operator Not Supported After $group

**Problem:** Using `$ne` operator in a `$match` stage that followed `$group` raised `MalformedQueryException: Operator '' is not a valid query operation`.

**Root Cause:** After `$group`, temporary tables have `id` and `data` columns (with `_id` embedded in JSON), but the WHERE clause generator referenced `_id` as a direct column that doesn't exist.

**Solution:** Enhanced `_process_match_stage()` to handle post-`$group` table schemas:

1. **Schema Detection** â€” Check if current table has `_id` column or only `data` column
2. **WHERE Clause Rewriting** â€” Replace `_id` column references with `json_extract(data, '$_id')` when needed
3. **SELECT Clause Adaptation** â€” Build appropriate SELECT based on available columns

**Impact:**
- **Before:** `$match` after `$group` with `$ne` â†’ `MalformedQueryException`
- **After:** `$match` after `$group` with `$ne` â†’ Works correctly

**Example:**

```python
# This now works!
pipeline = [
    {'$match': {'status': 'published'}},
    {'$unwind': '$tags'},
    {'$group': {'_id': '$tags', 'count': {'$sum': 1}}},
    {'$match': {'_id': {'$ne': 'excluded_tag'}}},  # Now supported!
]

results = list(db.posts.aggregate(pipeline))
# Returns all tags except 'excluded_tag'
```

**Workaround No Longer Needed:**

```python
# NO LONGER NEEDED - $ne after $group now works natively

# Old workaround (v1.14.3 and earlier):
results = list(db.posts.aggregate(pipeline_without_ne))
results = [r for r in results if r['_id'] != 'excluded_tag']

# New approach (v1.14.4+):
pipeline = [
    {'$match': {'status': 'published'}},
    {'$unwind': '$tags'},
    {'$group': {'_id': '$tags', 'count': {'$sum': 1}}},
    {'$match': {'_id': {'$ne': 'excluded_tag'}}},
]
results = list(db.posts.aggregate(pipeline))
# Already filtered correctly!
```

## Migration Guide

### Upgrading from v1.14.3

This is a non-breaking patch release. All existing code continues to work without modification.

### For Users Affected by the Bugs

#### BUG-4: $lookup Empty Array Type

If you worked around the string `"[]"` issue:

```python
# Before v1.14.4: Workaround needed
results = list(db.posts.aggregate(pipeline))
comments = results[0].get('comments', [])
if isinstance(comments, str):
    import json
    comments = json.loads(comments) if comments != '[]' else []

# After v1.14.4: Works natively
results = list(db.posts.aggregate(pipeline))
comments = results[0].get('comments', [])  # Always a list!
```

#### BUG-2: ObjectId Parameter Binding

If you converted ObjectId to string manually:

```python
# Before v1.14.4: Manual conversion required
pipeline = [{'$match': {'_id': str(oid)}}]

# After v1.14.4: Pass ObjectId directly
pipeline = [{'$match': {'_id': oid}}]
```

#### BUG-3: $lookup Type Mismatch

If you ensured consistent field types manually:

```python
# Before v1.14.4: Had to store parent_post as string to match string _id
db.comments.insert_one({'parent_post': str(post_oid)})

# After v1.14.4: ObjectId type coercion works automatically
db.comments.insert_one({'parent_post': post_oid})  # Works!
```

#### LIM-3: $ne After $group

If you filtered results in Python:

```python
# Before v1.14.4: Manual filtering
results = list(db.posts.aggregate(pipeline))
results = [r for r in results if r['_id'] != exclude_tag]

# After v1.14.4: Use $ne in pipeline
pipeline = [
    {'$group': {'_id': '$tags', 'count': {'$sum': 1}}},
    {'$match': {'_id': {'$ne': exclude_tag}}},
]
results = list(db.posts.aggregate(pipeline))
```
