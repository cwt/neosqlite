# NeoSQLite v1.14.15 Release Notes

## TL;DR

NeoSQLite v1.14.15 is a **bug-fix release** that addresses SQL injection vectors, data integrity issues, and correctness bugs identified since v1.14.14.

**Key Fixes:**
1. **SQL Injection via Collection Names** ŌĆö All SQL f-strings in cursor and raw batch cursor now pass collection names through `quote_table_name()`, and GridFS legacy migration does the same.
2. **Data Integrity** ŌĆö Fixed ChangeStream `_last_id` ordering (moved after commit), `__del__` accidental commit (now rolls back), and GridIn empty/exact-chunk MD5 (now always computed).
3. **Correctness** ŌĆö ChangeStream ensures `_id` column exists before creating triggers; `with_options()` clones get independent `_collections` dicts.
4. **New: `query_only` Command** ŌĆö `conn.command("query_only")` gets/sets SQLite read-only mode with int, bool, str, and dict value support.

- **Upgrade if you use:** collections with names containing SQL keyword substrings, GridFS with exact-chunk or empty files, Change Streams, `with_options()`, or read-only database access.
- **Impact:** Hardens security posture and fixes data integrity edge cases. No breaking API changes.

## Overview

This release is a focused patch addressing seven bugs and one minor feature discovered after v1.14.14. All changes are backward-compatible.

### Security Hardening

#### SQL Injection: Unquoted Collection Names in Cursors

**Severity: High.** Eight SQL f-strings in `cursor.py` and `raw_batch_cursor.py` used `self._collection.name` directly without quoting. A collection named `"from_parts"`, `"select_data"`, or similar would produce invalid SQL or inject unintended SQL fragments.

**Before:**
```python
sql = f"SELECT ... FROM {self._collection.name} WHERE ..."
```
**After:**
```python
from ..sql_utils import quote_table_name
sql = f"SELECT ... FROM {quote_table_name(self._collection.name)} WHERE ..."
```

#### SQL Injection: GridFS Legacy Migration

`_migrate_legacy_tables_if_needed()` in `gridfs_bucket.py` constructed new table names without `quote_table_name()`, inconsistent with how `_create_collections()` quotes names. Fixed to use `quote_table_name()` for both `new_files` and `new_chunks`.

### Data Integrity Fixes

#### ChangeStream `_last_id` Updated Before Commit

**Severity: High.** In both the normal event-processing path and the `skip_change` path, `_last_id` was assigned **before** `commit()`. If the commit failed (crash, disk full), `_last_id` would advance past the uncommitted row, causing that change event to be **skipped forever** on recovery.

**Fix:** Moved `_last_id = change_id` to **after** `commit()` in both paths, with clear ordering comments.

**Before:**
```
self._last_id = change_id    ŌåÉ advances BEFORE commit
self._collection.db.commit()
```
**After:**
```
self._collection.db.commit()
self._last_id = change_id    ŌåÉ advances only AFTER successful commit
```

#### `__del__` Accidental Commit During Garbage Collection

**Severity: Medium.** `Connection.__del__()` called `self.close()`, which commits any pending transaction. If a Connection was garbage collected mid-transaction (e.g., unhandled exception), partial work was accidentally persisted.

**Fix:** New `_close_rollback()` method that rolls back (not commits) pending transactions before closing. `__del__` now delegates to this method. Robust exception handling covers interpreter-shutdown scenarios where `sqlite3` may already be `None`.

#### GridIn `close()` Stores `None` for Empty-File / Exact-Chunk MD5

**Severity: Medium.** The metadata update (length, md5) in `GridIn.close()` was inside a conditional that required residual buffer data to be non-empty. When the last chunk exactly filled the buffer (empty buffer but chunks written), or the file was empty, the update was skipped entirely ŌĆö leaving `md5` and `length` as `NULL`.

**Fix:** Restructured `close()` to always update file metadata when a file document exists. Empty-file MD5 is now correctly computed as `hashlib.md5(b"").hexdigest()`.

### Correctness Fixes

#### ChangeStream Triggers Reference Missing `_id` Column

When watching a legacy collection created without an `_id` column, the ChangeStream's SQL triggers (`NEW._id` / `OLD._id`) would fail at runtime.

**Fix:** `ChangeStream.__init__()` now calls `_ensure_id_column_exists()` before `_setup_triggers()`.

#### `with_options()` Clones Share `_collections` Dict Mutably

`with_options()` assigned `clone._collections = self._collections` directly, so mutations (add/delete) to one Connection's collection cache silently affected the other.

**Fix:** Uses `self._collections.copy()` for independent dicts. `Collection` objects remain shared (correct behavior ŌĆö they wrap the same SQL table).

### New Feature: `query_only` Command

The `Connection.command()` method now supports `"query_only"` for reading and setting SQLite's `PRAGMA query_only`:

```python
# Get current state
conn.command("query_only")           # ŌåÆ {"ok": 1, "query_only": False}

# Set with int, bool, or str
conn.command("query_only", 1)        # enable
conn.command("query_only", True)     # enable
conn.command("query_only", "ON")     # enable
conn.command("query_only", False)    # disable

# Dict-style
conn.command({"query_only": 1})      # enable
```

Invalid values return `{"ok": 0, "errmsg": ...}`.

## Migration Guide

### Upgrading from v1.14.14

This is a non-breaking release. All existing code continues to work without modification.

**Recommended for all users**, especially those running:
- Collections with names containing SQL keywords (`from`, `select`, `where`, etc.)
- GridFS with empty files or files that exactly fill a chunk
- Change Streams where data loss on crash is unacceptable
- Applications using `with_options()` for multiple collection-access patterns
- Deployments needing read-only database access via `command("query_only")`
