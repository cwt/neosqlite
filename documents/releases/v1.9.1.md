## Overview

NeoSQLite v1.9.1 is a **maintenance and optimization release** that introduces configurable SQLite journal modes and significant cursor performance improvements. This version focuses on providing flexibility for specialized deployment environments while maintaining full PyMongo API compatibility.

**Key Achievement**: With this release, NeoSQLite provides **production-grade flexibility** for SQLite deployment scenarios, allowing developers to choose the optimal journal mode for their specific use case while maintaining ACID compliance and PyMongo API parity.

## Major Feature: Configurable Journal Mode

Full support for SQLite journal mode configuration via `Connection` constructor:

### Journal Mode Options

- ✅ **WAL** (Default) - Write-Ahead Logging for best concurrency
- ✅ **DELETE** - Traditional rollback journal, single-file when closed
- ✅ **TRUNCATE** - Similar to DELETE, but truncates journal
- ✅ **PERSIST** - Overwrites journal with zeros for reuse
- ✅ **MEMORY** - Journal in RAM, fastest but no crash recovery
- ✅ **OFF** - No journaling, fastest but no transaction safety

### API Usage

```python
from neosqlite import Connection, JournalMode

# Using enum (recommended)
db = Connection("app.db", journal_mode=JournalMode.WAL)

# Using string
db = Connection("app.db", journal_mode="DELETE")
```

### Type Safety

- **`JournalMode` class** - Enum-like class with validated constants
- **Case-insensitive** - Accepts "wal", "WAL", "Wal", etc.
- **Validation** - Raises `ValueError` for invalid modes
- **Clone propagation** - Journal mode correctly propagated to cloned connections

## Performance: SQL-Tier Cursor Optimization

Significant cursor performance improvements by moving operations to SQL level:

### Cursor Optimizations

**`sort()` Optimization**:
- Moved to SQL `ORDER BY` clause
- Eliminates Python-level sorting overhead
- Supports collation for case-insensitive sorting

**`limit()` and `skip()` Optimization**:
- Moved to SQL `LIMIT` and `OFFSET` clauses
- Prevents $O(N)$ memory overhead for large result sets
- Up to **2.4x performance improvement** in cursor benchmarks

**Automatic Tier Selection**:
- Smart detection of when SQL optimization is possible
- Falls back to Python when complex filters require it
- Transparent to users — no API changes needed

### Performance Impact

| Operation | Before | After | Improvement |
|-----------|--------|-------|-------------|
| **Cursor with sort** | Python sort | SQL ORDER BY | ~2.4x |
| **Cursor with limit** | Load all, slice | SQL LIMIT | Memory efficient |
| **Cursor with skip** | Load all, skip | SQL OFFSET | Memory efficient |
| **Combined operations** | Multiple passes | Single SQL query | Significant |

## Known Limitations

### Journal Mode Considerations

**WAL Mode**:
- Creates `-wal` and `-shm` files alongside database
- Not suitable for NFS or network filesystems
- Best concurrency for local deployments

**DELETE Mode**:
- Writers block readers (lower concurrency than WAL)
- Single file when database is closed
- Required for NFS deployments

**MEMORY/OFF Modes**:
- No crash recovery guarantees
- Use only for specific use cases (caching, temporary data)

### Cursor Optimization Limitations

**Python Fallback Required When**:
- Complex `$expr` filters that can't be translated to SQL
- `$where` predicates using Python functions
- Certain edge cases with nested array queries

**Performance Impact**:
- SQL-tier optimization provides 10-100x improvement for supported operations
- Python fallback still functional but slower
