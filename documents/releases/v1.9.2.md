## Overview

NeoSQLite v1.9.2 is a **performance-focused release** that significantly expands SQL-tier aggregation pipeline optimization and implements SQL-level optimizations for complex update operators. This version reduces reliance on Python fallback, delivering faster query execution and better resource utilization while maintaining full PyMongo API compatibility.

**Key Achievement**: With this release, NeoSQLite achieves **unprecedented SQL-tier coverage** for aggregation pipelines and update operations, moving critical performance paths from Python to native SQLite execution for maximum efficiency.

## Major Feature: Expanded SQL-Tier Aggregation Optimization

Comprehensive support for additional aggregation pipeline stages translated to native SQL:

### New SQL-Translated Stages

- ✅ **$unset** - Remove fields from documents
- ✅ **$replaceRoot** - Promote embedded document to root
- ✅ **$replaceWith** - Replace document with specified expression
- ✅ **$sample** - Randomly select documents
- ✅ **$bucket** - Group documents into buckets by expression
- ✅ **$bucketAuto** - Automatically bucket documents by expression
- ✅ **$redact** - Restrict content access based on criteria
- ✅ **$unionWith** - Combine results with another collection
- ✅ **$lookup** - Left outer join with another collection
- ✅ **$merge** - Merge aggregation results into collection

### New SQL Operators

**`$objectToArray`**:
- Convert objects to key-value arrays
- Native SQL translation for better performance
- Supports dynamic object field extraction

### $redact Stage Enhancements

**PRUNE/KEEP Pattern Support**:
- Optimized for both `KEEP/PRUNE` and `PRUNE/KEEP` conditional patterns
- Handles `then: $$PRUNE` (inverts to WHERE NOT condition)
- Automatic fallback to Python evaluation for DESCEND patterns

**Example**:

```python
# SQL-optimized $redact with PRUNE/KEEP pattern
db.collection.aggregate([
    {
        "$redact": {
            "$cond": {
                "if": { "$eq": ["$level", 5] },
                "then": "$$PRUNE",
                "else": "$$DESCEND"
            }
        }
    }
])
```

## Performance: SQL-Level Update Operator Optimization

Optimized update operations by implementing Tier-2 SQL paths for complex array operators:

### Optimized Update Operators

**`$pop`**:
- Remove first/last element from array
- Native SQL using `json_remove()` and array indexing
- Eliminates Python document fetch-modify-update cycle

**`$push` with `$each`**:
- Append multiple elements to array
- SQL-level array concatenation using `json_patch()`
- Supports `$slice` modifier for array size limits

**`$addToSet`**:
- Add element to array if not present
- Conditional SQL logic to check existence before insertion
- Optimized for both single values and `$each` batches

### SQLite Version Detection

**Automatic Feature Detection**:
- `_supports_relative_json_indexing()` detects SQLite 3.42.0+ capabilities
- Enables advanced JSON operations when available
- Graceful fallback for older SQLite versions

**Feature Cache**:
- `_SQLITE_FEATURES` cache for efficient capability checking
- Type-safe implementation for mypy compliance

## Known Limitations

### SQL-Tier Aggregation Limitations

**Python Fallback Required When**:
- Complex expressions can't be translated to SQL
- Custom JavaScript functions in $where
- Certain edge cases with deeply nested arrays
- DESCEND pattern in $redact (falls back to Python)

**Performance Impact**:
- SQL-tier optimization provides 10-100x improvement for supported stages
- Python fallback remains functional but slower

### Update Operator Limitations

**SQLite Version Dependencies**:
- Relative JSON indexing requires SQLite 3.42.0+
- Older versions use alternative SQL patterns
- Automatic detection and fallback handled transparently

**Complex Modifiers**:
- $push with multiple modifiers may require Python fallback
- $addToSet with complex objects uses SQL with limitations

### $redact Stage Limitations

**DESCEND Pattern**:
- Falls back to Python evaluation for recursive descent
- SQL optimization only for PRUNE/KEEP terminal patterns

**Complex Conditions**:
- Nested $redact stages may not fully optimize
- Mixed patterns require careful analysis
