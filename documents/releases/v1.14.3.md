# NeoSQLite v1.14.3 Release Notes

## TL;DR

NeoSQLite v1.14.3 is a **bug fix and performance optimization release** that resolves SQL errors when using `$addFields` with complex expressions (like `$filter`, `$map`, `$reduce`) after `$lookup` in aggregation pipelines.

**Critical Fix:** This release eliminates the `sqlite3.OperationalError: no such column: _id` error and implements a hybrid Python approach that keeps pipelines in Tier 2 (temporary tables) instead of falling back to Tier 3 (full Python), delivering **10-100x better performance** for complex aggregation pipelines.

- **Upgrade if you use:** Aggregation pipelines with `$addFields` containing `$filter`, `$map`, `$reduce`, or other array transformation operators after `$lookup`.
- **Result Correctness:** Achieved **100% compatibility** in API comparison reports (386 tests, 0 failures).

## High Severity Fixes

### Fixed $addFields After $lookup SQL Error

**Problem:** When using `$addFields` with complex expressions (like `$filter`, `$map`, `$reduce`) after `$lookup` in an aggregation pipeline, NeoSQLite threw `sqlite3.OperationalError: no such column: _id`. This occurred because temporary tables created by `$lookup` only have `id` and `data` columns, but `$addFields` assumed `_id` existed as a top-level column.

**Performance Impact:** Previously, this error caused the entire pipeline to fall back to Tier 3 (full Python), re-loading all documents from the original collection and re-running all stages in Python — throwing away the SQL work already completed.

**Solution:** Implemented a three-part fix:

1. **Column Schema Detection** — `_process_add_fields_stage` now checks which columns actually exist in the current temp table before building SELECT clauses, preventing SQL errors.

2. **Hybrid Python Approach** — Added `_process_add_fields_stage_python_hybrid` method that loads documents from the current temp table, processes complex expressions in Python using `ExprEvaluator`, and creates a new temp table. This keeps pipelines in **Tier 2** instead of falling back to Tier 3.

3. **Variable Navigation Fix** — Fixed `$$variable.field.path` syntax evaluation in `_evaluate_operand_python` to properly split variable references and navigate into variable values (e.g., `$$comment.comment_author` now correctly extracts the `comment_author` field from the `$$comment` variable).

**Performance Impact:**
- **Before:** `$addFields` with `$filter` → Tier 3 fallback (re-runs entire pipeline in Python)
- **After:** `$addFields` with `$filter` → Tier 2 hybrid (keeps SQL work, only processes complex stage in Python)

**Example:**

```python
# This now works correctly and stays in Tier 2!
pipeline = [
    {'$match': {'_id': 1}},
    {'$lookup': {
        'from': 'comments',
        'localField': '_id',
        'foreignField': 'post_id',
        'as': 'comments'
    }},
    {'$addFields': {
        'comments': {
            '$filter': {
                'input': '$comments',
                'as': 'comment',
                'cond': {'$in': ['$$comment.author', ['alice', 'bob']]}
            }
        }
    }}
]

results = list(db.posts.aggregate(pipeline))
# Returns correctly filtered results
```

## Medium Severity Enhancements

### Code Quality & Modernization

**Python 3.9+ Type Hints** — Replaced deprecated `typing.List`, `typing.Dict`, and `typing.Tuple` with built-in generics (`list`, `dict`, `tuple`) across the codebase for cleaner, more modern Python code.

**F-string Debug Logging** — Migrated debug logging statements to use Python 3.8+ f-string `=` specifier for more concise and readable variable logging (e.g., `f"{value=}"` instead of `f"value={value}"`).

**Removed Duplicate Compound Index Creation** — Fixed bug in `index_manager.py` where compound indexes were being created twice, wasting database resources and potentially causing conflicts.

**Walrus Operator Refactoring** — Simplified `cursor.fetchone()` patterns throughout the codebase using the walrus operator (`:=`) for more concise and Pythonic code.

**GridFS Bucket Name Extraction** — Replaced manual string slicing with `str.removesuffix()` for clearer intent and reduced error potential.

## Migration Guide

### Upgrading from v1.14.2

This is a non-breaking patch release. All existing code continues to work without modification.

### For Users Affected by the Bug

If you encountered `sqlite3.OperationalError: no such column: _id` when using `$addFields` with `$filter`, `$map`, or `$reduce` after `$lookup`, this release completely resolves your issue:

```python
# Before v1.14.3: This would fail with SQL error or return incorrect results
# After v1.14.3: This works correctly and efficiently

pipeline = [
    {'$match': {'_id': post_id}},
    {'$lookup': {
        'from': 'comments',
        'localField': '_id',
        'foreignField': 'post_id',
        'as': 'comments'
    }},
    {'$addFields': {
        'comments': {
            '$filter': {
                'input': '$comments',
                'as': 'comment',
                'cond': {'$in': ['$$comment.author', active_users]}
            }
        }
    }}
]

results = list(db.posts.aggregate(pipeline))
# Now returns correctly filtered comments
```

### Workaround No Longer Needed

The Python post-processing workaround mentioned in bug reports is no longer necessary:

```python
# NO LONGER NEEDED - NeoSQLite now handles this natively

# Old workaround (v1.14.2 and earlier):
pipeline = [
    {'$match': {'_id': post_id}},
    {'$lookup': {...}}
]
results = list(db.posts.aggregate(pipeline))
post = results[0]
comments = [c for c in post['comments'] if c['author'] in active_users]

# New approach (v1.14.3+):
pipeline = [
    {'$match': {'_id': post_id}},
    {'$lookup': {...}},
    {'$addFields': {
        'comments': {
            '$filter': {...}
        }
    }}
]
results = list(db.posts.aggregate(pipeline))
# Comments already filtered in results
```
