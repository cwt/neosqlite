## Major Feature: AutoVacuum Support with Transparent Migration

### The Problem

SQLite's `auto_vacuum` setting controls how disk space is reclaimed after deletions. Without proper configuration, deleted data leaves "free space" in the database file but never shrinks the file itself — potentially wasting gigabytes of disk space.

### The Solution

NeoSQLite v1.12.0 provides:

- **AutoVacuumMode Class**: NONE (0), FULL (1), INCREMENTAL (2) modes
- **Connection Parameter**: `auto_vacuum` accepts int or string
- **Safe Migration**: Opt-in migration for existing databases via `AUTOVACUUM_MIGRATION=1`

### AutoVacuum Modes

| Mode | Behavior | Best For |
|------|----------|----------|
| **NONE** | Never reclaim space automatically | Read-only databases, temporary databases |
| **FULL** | Automatically vacuum after deletions | Frequent deletes where disk space is critical |
| **INCREMENTAL** | Track freed pages, vacuum on demand | **Most applications** (default) |

### Safe Migration Flow

```text
┌─────────────────────────────────────────────────────────────┐
│  1. Check if current auto_vacuum matches requested mode     │
│     └─ If same: Skip migration (no changes made)            │
└─────────────────────────────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────────┐
│  2. Checkpoint WAL files (ensure all data is in main file)  │
└─────────────────────────────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────────┐
│  3. Create timestamped backups of:                          │
│     - database.db                                           │
│     - database.db-wal (if exists)                           │
│     - database.db-shm (if exists)                           │
└─────────────────────────────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────────┐
│  4. Open backup, set new auto_vacuum, VACUUM INTO new file  │
└─────────────────────────────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────────┐
│  5. Replace original database with new vacuumed file        │
└─────────────────────────────────────────────────────────────┘
                            │
                            ▼
┌─────────────────────────────────────────────────────────────┐
│  6. Clean up WAL files and backups                          │
│     ✅ Migration complete!                                  │
└─────────────────────────────────────────────────────────────┘
```

### Safety Features

- **Opt-in**: Migration only runs when explicitly enabled
- **No-op when matching**: If modes already match, file is never touched
- **Full backup**: All related files backed up before changes
- **Atomic replacement**: Uses temp file + move operation
- **Rollback on error**: If anything fails, original database is restored

### Usage

```python
import neosqlite

# Default: INCREMENTAL mode for new databases
conn = neosqlite.Connection("myapp.db")

# Explicit mode using class (recommended)
conn = neosqlite.Connection(
    "myapp.db",
    auto_vacuum=neosqlite.AutoVacuumMode.FULL
)

# Migrate existing database (requires env var)
import os
os.environ["AUTOVACUUM_MIGRATION"] = "1"
conn = neosqlite.Connection("existing.db", auto_vacuum=neosqlite.AutoVacuumMode.INCREMENTAL)
```

## Major Feature: MongoDB-Compatible compact Command

NeoSQLite implements MongoDB-compatible `compact` with SQLite-specific extensions:

```python
# Compact with default threshold (20MB) - uses incremental vacuum
result = conn.command("compact", "collection_name")
# Returns: {'bytesFreed': 12345, 'ok': 1}

# Dry run - estimate without actually compacting
result = conn.command("compact", "collection_name", dryRun=True)
# Returns: {'estimatedBytesFreed': 12345, 'ok': 1}

# With custom threshold - uses incremental vacuum in batches
result = conn.command("compact", "collection_name", freeSpaceTargetMB=10)
# Only runs if free space >= 10MB, uses incremental vacuum with 10MB batches

# Full vacuum (no incremental)
result = conn.command("compact", "collection_name", freeSpaceTargetMB=0)
# Runs full VACUUM
```

### freeSpaceTargetMB Explained

NeoSQLite extends MongoDB's `freeSpaceTargetMB` parameter to serve **three purposes**:

#### 1. Threshold

Only run compaction if free space >= `freeSpaceTargetMB`:

```python
# Only compact if free space >= 20MB (MongoDB default)
result = conn.command("compact", "collection", freeSpaceTargetMB=20)
# If free space < 20MB: {'bytesFreed': 0, 'ok': 1}
```

This is incremental VACUUM.

#### 2. Batch Size

This value also used as the batch size:

```python
# freeSpaceTargetMB=10 means:
#   - Threshold: only run if free >= 10MB
#   - Batch: vacuum 10MB worth of pages per iteration

result = conn.command("compact", "collection", freeSpaceTargetMB=10)
# Internally: loops incremental_vacuum() with 10MB batches until all free pages reclaimed
```

#### 3. Full VACUUM

If set `freeSpaceTargetMB=0`, VACUUM runs in full mode.

```python
result = conn.command("compact", "collection", freeSpaceTargetMB=0)
# Internally: db.execute("VACUUM")
```

### Behavior Matrix

| Value | Behavior |
|-------|----------|
| Not specified | Incremental vacuum with 20MB threshold and batch size (MongoDB default) |
| `freeSpaceTargetMB=10` | Incremental vacuum with 10MB threshold and batch size |
| `freeSpaceTargetMB=0` | Full VACUUM |

### Why Extend?

- **MongoDB compatibility**: Default 20MB threshold matches MongoDB behavior
- **SQLite performance**: SQLite incremental VACUUM is lightweight and non-blocking

## Major Feature: dbStats Command

### The Problem

Applications need database statistics for monitoring and capacity planning. MongoDB provides `dbStats` for this purpose.

### The Solution

NeoSQLite implements MongoDB-compatible `dbStats`:

```python
result = conn.command("dbStats")
# Returns:
# {
#     "ok": 1,
#     "db": "mydb",
#     "collections": 5,
#     "views": 0,
#     "objects": 10000,
#     "avgObjSize": 256,
#     "dataSize": 2560000,
#     "storageSize": 4096000,
#     "indexes": 7,
#     "indexSize": 512000,
#     "totalSize": 4608000,
#     "fsTotalSize": 500000000000,
#     "fsUsedSize": 250000000000,
#     "scaleFactor": 1
# }
```

### Key Improvements

- **Accurate Index Sizes**: Uses SQLite's `dbstat` virtual table for actual index sizes (v1.12.0 improvement over percentage estimates)
- **Filesystem Stats**: `fsTotalSize` and `fsUsedSize` via `shutil.disk_usage()`
- **Zero for In-Memory**: Returns 0 for `:memory:` databases (no filesystem)

## Additional Maintenance Commands

### WAL Checkpoint

Control SQLite's Write-Ahead Log checkpoint behavior:

```python
# Default PASSIVE mode - checkpoints if possible, don't block
result = conn.command("wal_checkpoint")
# Returns: {'ok': 1, 'mode': 'PASSIVE', 'busy': 0, 'log': 100, 'checkpointed': 50}

# FULL - checkpoint and block until complete
result = conn.command("wal_checkpoint", mode="FULL")

# TRUNCATE - checkpoint and truncate WAL file
result = conn.command("wal_checkpoint", mode="TRUNCATE")
```

### Cache Size

Control SQLite's in-memory page cache:

```python
# Get current cache size (negative = KB)
result = conn.command("cache_size")
# Returns: {'ok': 1, 'cache_size': -2000}  # 2MB cache

# Set cache size
result = conn.command("cache_size", pages=1000)
# Returns: {'ok': 1, 'message': 'cache_size set to 1000'}

# Common values: -2000 (2MB), -4000 (4MB), -8000 (8MB)
```

### Busy Timeout

Configure lock wait duration:

```python
# Get current timeout (milliseconds)
result = conn.command("busy_timeout")
# Returns: {'ok': 1, 'busy_timeout': 5000}  # 5 seconds

# Set timeout
result = conn.command("busy_timeout", milliseconds=5000)

# Common values: 0 (immediate), 1000 (1s), 5000 (5s), 30000 (30s)
```

## Known Limitations

### AutoVacuum Migration

**Opt-in Requirement**: Migration is disabled by default. This prevents unexpected file modifications but requires explicit opt-in for migration.

**Large Databases**: Migration temporarily requires ~2x database size in free disk space. For very large databases (>10 GB), schedule migration during maintenance windows.

### compact Command

**SQLite Scope**: Unlike MongoDB which compacts per-collection, SQLite compacts the entire database. The collection name parameter is accepted for API compatibility but has no effect.

**Fragmentation**: Incremental vacuum only reclaims pages from the *end* of the database file. Internal fragmentation requires full VACUUM.

### dbStats

**In-Memory Databases**: Returns 0 for filesystem-related stats (fsTotalSize, fsUsedSize) since `:memory:` databases have no filesystem presence.
