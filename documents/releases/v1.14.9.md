# NeoSQLite v1.14.9 Release Notes

## TL;DR

NeoSQLite v1.14.9 is a **bug fix release** that resolves two critical aggregation pipeline bugs and improves `$expr` operator compatibility with MongoDB.

**Key Fixes:**
1. **`$elemMatch` in `$match` stage** — Aggregation pipelines now correctly filter documents using `$elemMatch`, matching `find()` behavior. Previously returned unfiltered results.
2. **`$in` on array fields in `$match` stage** — Aggregation pipelines now properly handle `$in` operator on array fields via Python fallback. Previously returned 0 results.
3. **`$size` single operand in `$expr`** — The MongoDB-compatible syntax `{"$size": "$field"}` (without list wrapper) now works correctly in `$expr` contexts.

- **Upgrade if you use:** `$elemMatch` or `$in` in aggregation `$match` stages, or `$size` with single field reference in `$expr` queries.
- **Result Correctness:** Both `$elemMatch` and `$in` now return correct results matching `find()` behavior — no more false positives or empty results.

---

## Overview

This release fixes three bugs across two severity levels:

- **$elemMatch Aggregation Bug** — `$match` stage with `$elemMatch` returned unfiltered documents instead of applying proper array element matching.
- **$in Array Fields Aggregation Bug** — `$match` stage with `$in` on array fields returned 0 results due to incorrect SQL translation.
- **$size Single Operand** — MongoDB-compatible `{$size: "$field"}` syntax raised `ValueError` in `$expr` contexts.

## High Severity Fixes

### Fixed $elemMatch Returning Unfiltered Results in Aggregation

**Bug:** `$match` stage with `$elemMatch` (e.g., `{"tags": {"$elemMatch": "value"}}`) returned all documents instead of filtering by array element match when processed through Tier 2 (temporary table aggregation).

**Error Symptom:**

```python
# find() - correct (returns 3 posts)
r1 = list(collection.find({"tags": {"$elemMatch": "maliit-keyboard"}}))

# aggregate() - WRONG (returns 8 posts, all unfiltered)
r2 = list(collection.aggregate([
    {"$match": {"tags": {"$elemMatch": "maliit-keyboard"}}}
]))
```

**Root Cause:** The temporary table aggregation (`_process_match_stage()`) treated SQL translation failures (`where_clause is None`) as "text search involved" and created an unfiltered temporary table with ALL documents. However, `where_clause is None` also occurs when SQL translation fails for unsupported operators like `$elemMatch`.

**Solution:** When `where_clause is None`, check if it's actually text search (using `_contains_text_search()`). If not text search, raise `NotImplementedError` to trigger the Python fallback which correctly handles `$elemMatch` via `_elemMatch` in `query_operators.py`:

```python
# Before: Always returned unfiltered table when where_clause is None
if where_clause is None:
    new_table = create_temp(match_stage, f"SELECT * FROM {current_table}")
    return new_table

# After: Distinguish text search from unsupported operators
if where_clause is None:
    if _contains_text_search(match_spec):
        # Text search: return unfiltered for special handling
        new_table = create_temp(match_stage, f"SELECT * FROM {current_table}")
        return new_table
    else:
        # Unsupported operator: trigger Python fallback
        raise NotImplementedError(
            f"$match stage contains unsupported operators: {match_spec}"
        )
```

**Impact:**
- **Before:** `$elemMatch` in `$match` → all documents returned → incorrect results
- **After:** Properly falls back to Python filtering → correct array element matching

**Affected Syntax Variants:**
- `{"$elemMatch": "value"}` — ✅ Now works correctly
- `{"$elemMatch": {"$eq": "value"}}` — ✅ Now works correctly
- `{"$elemMatch": {"$gte": 90}}` — ✅ Already worked (no change)

### Fixed $in on Array Fields Returning 0 Results in Aggregation

**Bug:** `$match` stage with `$in` on array fields (e.g., `{"tags": {"$in": ["value"]}}`) returned 0 results when processed through Tier 2 (temporary table aggregation).

**Root Cause:** The SQL translation of `$in` generates a simple `IN` clause:
```sql
json_extract(data, '$.tags') IN ('value')
```

This doesn't work for array fields because `json_extract` returns the entire JSON array as a string, and comparing this string against scalar values never matches. The proper array-aware logic exists in Python filtering (`_in()` function in `query_operators.py`) but Tier 2 wasn't falling back to it.

**Solution:** Same fix as `$elemMatch` — when SQL translation returns `None`, raise `NotImplementedError` to trigger Python fallback which has correct array semantics:

```python
def _in(field: str, value: list[Any], document: dict[str, Any]) -> bool:
    doc_value = _get_nested_field(field, document)
    if isinstance(doc_value, list):
        return any(item in value for item in doc_value)  # Array-aware
    else:
        return doc_value in value
```

**Impact:**
- **Before:** `$in` on array fields → 0 results (SQL comparison fails)
- **After:** Falls back to Python → correct array element matching

## Medium Severity Fixes

### Fixed $size Single Operand in $expr Contexts

**Bug:** The MongoDB-compatible syntax `{"$expr": {"$gt": [{"$size": "$tags"}, 1]}}` raised `ValueError: $size requires exactly 1 operand` when `$size` received a single string operand (`"$tags"`) instead of a list-wrapped operand (`["$tags"]`).

**Error Message:**

```text
ValueError: $size requires exactly 1 operand
```

**Root Cause:** The Python evaluator (`_evaluate_array_python` in `python_evaluators.py`) was missing operand normalization for array operators. When operands is a string like `"$tags"`, `len("$tags")` returns 5 (string length), not 1 (operand count).

**Solution:** Added operand normalization matching the SQL converter's existing behavior:

```python
# Before: No normalization
def _evaluate_array_python(self, operator, operands, document):
    match operator:
        case "$size":
            if len(operands) != 1:  # len("$tags") == 5 → FAILS
                raise ValueError("$size requires exactly 1 operand")

# After: Normalize single operand to list
def _evaluate_array_python(self, operator, operands, document):
    if operator in ("$size", "$isArray", "$sum", "$avg", "$min", "$max") and not isinstance(operands, list):
        operands = [operands]
    
    match operator:
        case "$size":
            if len(operands) != 1:  # len(["$tags"]) == 1 → PASSES
                raise ValueError("$size requires exactly 1 operand")
```

**Impact:**
- **Before:** `{"$size": "$field"}` → `ValueError`
- **After:** `{"$size": "$field"}` → works correctly (normalized to `["field"]`)

**MongoDB Compatibility:** Both syntax variants now work:
```python
# List-wrapped (already worked)
{"$expr": {"$gt": [{"$size": ["$tags"]}, 1]}}

# Single operand (now fixed)
{"$expr": {"$gt": [{"$size": "$tags"}, 1]}}
```
