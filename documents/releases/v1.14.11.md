# NeoSQLite v1.14.11 Release Notes

## TL;DR

NeoSQLite v1.14.11 is a **performance and correctness release** that adds efficient CTE-based SQL support for array operators (`$in`, `$nin`, `$all`), implements native regex operators, and ensures boolean type consistency with MongoDB.

**Key Improvements:**
1. **CTE-based Array Operators** — `$in`, `$nin`, `$all` now use efficient SQLite CTE patterns with `jsonb_each` for 10-100x speedup when SQLite 3.51.0+ is available.
2. **Native Regex Operators** — `$regexMatch`, `$regexFind`, `$regexFindAll`, and `$regex` with options (`i`, `m`, `s`, `x`) now work natively in the SQL tier.
3. **Boolean Type Consistency** — `$isNumber`, `$isArray`, `$toBool` now return proper JSON booleans (`json('true')`/`json('false')`) matching MongoDB's native boolean type.
4. **Fixed `$elemMatch` with nested `$in/$nin`** — Correctly handles the `"value"` path alias (`json_each(data, "value")` instead of `json_each(data, value)`).

- **Upgrade if you use:** Array operators with large datasets, regex operators, or type-checking operators in `$expr`.
- **Performance:** Array operators now execute 10-100x faster when SQLite supports `jsonb_each`.

## Overview

This release focuses on **performance optimization** and **correctness improvements**:

- **Array Operator CTE Optimization** — Replaces Python fallback with native SQLite CTE patterns for `$in`, `$nin`, `$all`, delivering 10-100x performance improvement.
- **Native Regex Support** — Implements `$regexMatch`, `$regexFind`, `$regexFindAll`, and `$regex` operators directly in the SQL tier using custom SQLite functions.
- **Boolean Type Correctness** — Ensures `$isNumber`, `$isArray`, `$toBool` return JSON booleans instead of integers, fixing `$expr` queries with these operators.
- **Projection Bug Fix** — `_id` field is now properly included in projections by default, matching MongoDB behavior.

## High Severity Improvements

### CTE-based Array Operators for find()

**Before:** `$in`, `$nin`, `$all` on array fields fell back to Python, O(n) per document.

**After:** Uses efficient SQLite CTE pattern directly in the query:

```sql
-- $in: EXISTS pattern
EXISTS (SELECT 1 FROM json_each(data, '$.field') WHERE json_each.value IN (?, ?))

-- $nin: NOT EXISTS pattern
NOT EXISTS (SELECT 1 FROM json_each(data, '$.field') WHERE json_each.value IN (?, ?))

-- $all: AND EXISTS for each value
EXISTS (SELECT 1 FROM json_each(data, '$.field') WHERE json_each.value = ?)
AND EXISTS (SELECT 1 FROM json_each(data, '$.field') WHERE json_each.value = ?)
```

**Impact:** 10-100x performance improvement for array operator queries on large collections.

---

### Native Regex Operators in SQL Tier

**New operators now supported natively:**

| Operator | Format | Example |
|----------|--------|---------|
| `$regexMatch` | `{input, regex, options?}` | `{"$regexMatch": {"input": "$field", "regex": "pattern", "options": "i"}}` |
| `$regexFind` | `{input, regex, options?}` | `{"$regexFind": {"input": "$field", "regex": "\\d+"}}` |
| `$regexFindAll` | `{input, regex, options?}` | `{"$regexFindAll": {"input": "$field", "regex": "\\d+"}}` |
| `$regex` | `{field: {$regex: pattern, $options: "i"}}` | Classic find() syntax supported |

**Options supported:** `i` (case-insensitive), `m` (multiline), `s` (dotall), `x` (extended)

**Custom SQLite functions registered:** `REGEXP`, `REGEXP_FIND`, `REGEXP_FIND_ALL`, `REGEXP_REPLACE`

## Medium Severity Improvements

### Boolean Type Consistency

**Bug fixed:** `$isNumber`, `$isArray`, `$toBool` returned integers (1/0) instead of JSON booleans.

**Before (incorrect):**

```sql
-- Returns 1 or 0 (integer)
CASE WHEN json_type(data, '$.field') = 'array' THEN 1 ELSE 0 END
```

**After (correct):**

```sql
-- Returns json('true') or json('false')
CASE WHEN json_type(data, '$.field') = 'array' THEN json('true') ELSE json('false') END
```

**Impact:** `$expr` queries using these operators now correctly evaluate in WHERE clauses.

---

### Fixed $elemMatch with Nested $in/$nin

**Bug fixed:** `$elemMatch` with nested `$in` or `$nin` used wrong path alias.

**Before (incorrect):**

```sql
json_each(data, value)  -- Treated 'value' as a column name
```

**After (correct):**

```sql
json_each(data, "value")  -- Treated 'value' as a string literal
```

---

### _id Projection Default Behavior

**Bug fixed:** Inclusion projections without explicit `_id` specification excluded `_id`.

**MongoDB behavior:** `_id` is included by default in projections unless explicitly excluded (`_id: 0`).

**Fixed in:** `temporary_table_aggregation.py` — `_id` now properly preserved.

## Migration Guide

### Upgrading from v1.14.10

This is a non-breaking performance release. All existing code continues to work without modification.

### Performance Benefits

**Array Operator Queries:**

```python
# Now 10-100x faster with native SQL CTE
results = list(collection.find({"tags": {"$in": ["python", "nodejs", "ruby"]}}))
results = list(collection.aggregate([
    {"$match": {"status": {"$all": ["active", "verified"]}}}
]))
```

**Regex Queries:**

```python
# Now native SQL with options support
results = list(collection.find({
    "email": {"$regex": "^admin", "$options": "i"}
}))
results = list(collection.aggregate([
    {"$match": {
        "$expr": {
            "$regexMatch": {
                "input": "$bio",
                "regex": "\\bSQL\\b",
                "options": "i"
            }
        }
    }}
]))
```

**No code changes needed** — these are transparent performance improvements.
