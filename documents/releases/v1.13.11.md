# NeoSQLite v1.13.11 Release Notes

## TL;DR

NeoSQLite v1.13.11 is a **bug fix and code quality release** that resolves critical JSONB function mismatches affecting sort and min/max clause builders, fixes multiple SQL generation errors in temporary table aggregation pipelines, adds comprehensive debug logging for observability, and modernizes code with Python 3.10 pattern matching.

**Important:** The bugs fixed in this release **did not affect result correctness**. When SQL generation failed, NeoSQLite's tiered architecture automatically fell back to the Python implementation, ensuring results were always correct. These fixes improve performance by keeping more operations in the fast SQL tier.

- **Upgrade if you use:** Aggregation pipelines with `$setWindowFields`, `$graphLookup`, `$bucket`, `$bucketAuto`, or any operations on databases with/without JSONB support (for better performance)
- **No need to upgrade if:** You're on v1.13.10 and don't use aggregation pipelines or JSONB features
- **Breaking changes:** None — all existing code continues to work

---

## Breaking Changes

**None** — v1.13.11 is fully backward compatible with v1.13.10.

## Overview

This release focuses on correctness, observability, and code quality:

**Important Note on Result Correctness:** The SQL generation bugs fixed in this release **did not affect result correctness**. NeoSQLite's tiered architecture automatically falls back to the Python implementation when SQL generation fails. This means all results were already correct — these fixes improve **performance** by keeping more operations in the fast SQL tier rather than falling back to Python.

- **JSONB function mismatch fixed** — 3 places hardcoded wrong JSON functions regardless of JSONB support, now correctly select the appropriate function (improves SQL tier hit rate)
- **Temporary table aggregation SQL errors fixed** — Multiple SQL generation issues affecting `$bucket`, `$bucketAuto`, `$group`, `$setWindowFields`, `$graphLookup`, and window functions (all previously fell back to Python correctly)
- **Comprehensive debug logging** added across the codebase for improved observability and troubleshooting
- **Python 3.10 pattern matching** modernization across 7+ files, replacing long if/elif chains with cleaner match/case statements
- **Code deduplication** — Extracted `json_data_column` helper to eliminate 21 duplicated if/else blocks across 6 files
- **Test coverage improvements** — 60+ new tests added for find_operations, schema modules, and temporary table aggregation fixes

## High Severity Bug Fixes

### Fixed: JSONB-aware functions for sort and min/max clause builders

**Problem:** Three places hardcoded `json_extract`/`jsonb_extract` instead of selecting the correct function based on `_jsonb_supported`:

- **find_operations.py**: 3 inline sort builders used `json_extract` regardless of JSONB support
- **raw_batch_cursor.py**: Sort builder used `json_extract` regardless of JSONB support
- **cursor.py**: `_build_minmax_clause` used `jsonb_extract` regardless of JSONB support

**After:** All three now delegate to helpers that correctly use `self._json_function_prefix` or `_get_json_function_prefix(jsonb)` to select the appropriate function.

### Fixed: SQL errors in temporary table aggregation pipeline

**Problem:** Multiple SQL generation issues in `temporary_table_aggregation.py`:

- **_get_results_from_table**: Missing column detection for non-standard tables (e.g., `$bucket`, `$bucketAuto`, `$group` output)
- **_process_sort_skip_limit_stage**: Failed when only `id` column exists for `_id` sorting
- **_process_union_with_stage**: Missing column matching between unioned tables
- **_process_set_window_fields_stage**: Failed to handle `_id` as separate column in JSON operations
- **_build_window_frame_sql**: Incorrect window frame SQL (used `UNBOUNDED` instead of `UNBOUNDED PRECEDING`/`UNBOUNDED FOLLOWING`)
- **_process_graph_lookup_stage**: `restrictSearchWithMatch` failed due to leading "WHERE " and parameter count mismatch
- **_process_bucket_stage**: Returned incorrect `_id` values (now uses lower boundary value, MongoDB-compatible)
- **_process_bucket_auto_stage**: Missing `{min, max}` object structure
- **_process_group_stage**: Broken `$first`/`$last` accumulators

**After:** All issues resolved with proper column detection, window frame syntax, parameter handling, and MongoDB-compatible output formats.

## Medium Severity Bug Fixes

### Fixed: Silent exception handlers in error handling and fallback scenarios

**Problem:** Multiple exception handlers across the codebase silently swallowed errors without any logging, making troubleshooting difficult.

**Fix:** Added structured debug logging to all major exception handlers:

- **_sqlite.py, objectid.py, query_operators.py, connection.py**: Logger initialization and debug logging
- **changestream.py, gridfs/*, collection/**: Debug logs capturing error details before fallback behavior
- **Schema migration failures, index creation issues, type conversion errors**: Now logged with appropriate context
- **Query operators, JSON parsing, ObjectId handling**: Improved error context with exception details

All logs use debug level to avoid production noise while providing detailed diagnostic information during development and debugging sessions.

## Code Modernization: Python 3.10 Pattern Matching

Multiple modules have been refactored to use Python 3.10 `match/case` pattern matching:

### expr_evaluator.py

`$dateDiff` and `$datePart` date unit dispatch now uses match/case:

```python
# Before
if unit == "year":
    ...
elif unit == "month":
    ...
elif unit == "day":
    ...
# ... 6 more units

# After
match unit:
    case "year":
        ...
    case "month":
        ...
    case "day":
        ...
```

### temporary_table_aggregation.py

`$bucket` and `$bucketAuto` accumulator output now uses match/case.

### sql_tier_aggregator.py

Comparison operator dispatch (`$gt`, `$lt`, `$gte`, `$lte`, `$eq`, `$ne`) now uses match/case.

### query_engine/__init__.py

`$merge` stage `whenMatched`/`whenNotMatched` handling now uses match/case.

### Additional Refactoring (7 files total)

- **_matches_query_operators**: 8 comparison operators → `match op:`
- **$fill date unit dispatch**: 6 units → `match unit:`
- **_convert_to_bitmask**: `isinstance` int/list/tuple/iterable → `match value:`
- **_expression_uses_root**: `isinstance` str/dict/list → `match obj:`
- **search_in_value** (text_search, query_builder): `isinstance` str/dict/list → `match value:`
- **_apply_write_concern**: `isinstance` WriteConcern/dict → `match write_concern:`

## Code Deduplication: json_data_column Helper

**Problem:** The pattern:

```python
if self._jsonb_supported:
    ...json(data) as data...
else:
    ...data...
```

appeared **21 times across 6 files**.

**After:** Replaced with a single helper in `jsonb_support.py`:

```python
def json_data_column(jsonb_supported: bool, source: str = "data") -> str:
    return f"json({source})" if jsonb_supported else source
```

Each call site now uses a single f-string instead of a 4-line if/else.

## Migration Guide

### Upgrading from v1.13.10

**No breaking changes** — v1.13.11 is fully backward compatible.

### If You Use Aggregation Pipelines

The temporary table aggregation pipeline now correctly handles:

- `$bucket` and `$bucketAuto` output formats (MongoDB-compatible `_id` structures)
- `$setWindowFields` with proper window frame SQL syntax
- `$graphLookup` with `restrictSearchWithMatch` filtering
- `$group` with `$first`/`$last` accumulators

### If You Use JSONB Features

Sort and min/max clause builders now correctly select JSON functions based on whether your SQLite supports JSONB. This fixes potential errors on certain SQLite versions/configurations.

## Known Limitations

### NX-27017 Limitations

- **No replication/sharding**: Coming never™ — This is NX-class, not NCC-1701!
- **Change stream persistence**: Changes are tracked in memory; restart server = lost streams
- **`find_raw_batches` with batch_size**: Requires cursor state management
