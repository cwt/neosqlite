# NeoSQLite v1.14.0 Release Notes

## TL;DR

NeoSQLite v1.14.0 is a **major performance and API alignment release** that introduces native SQL-tier support for the `$project` aggregation stage and complex `_id` field operators. It also brings a breaking change to the compound index API to strictly align with PyMongo's `(field, direction)` tuple format, ensuring better compatibility with the standard MongoDB ecosystem.

**Breaking changes:** `create_index` and `create_indexes` now strictly require the PyMongo tuple format `[("field", 1)]` for compound indexes. Simple lists of strings (e.g., `["f1", "f2"]`) are no longer supported for compound indexes.

**Important Performance Gain:** By migrating `$project` and `_id` filtering from Python fallbacks directly into the SQL layer, this release significantly reduces data transfer overhead and processing time for large-scale queries and aggregation pipelines.

- **Upgrade if you use:** Aggregation pipelines with `$project`, complex queries on the `_id` field, or if you want closer alignment with PyMongo's indexing API.
- **Result Correctness:** 100% bit-for-bit result parity has been verified between the new SQL-tier paths and the original Python fallbacks.

---

## Breaking Changes

### Compound Index API Alignment

To align with PyMongo 4.x and support sort directions in compound indexes, the `create_index` and `create_indexes` methods now strictly require the tuple-based format for compound keys.

**Before (v1.13.11):**

```python
collection.create_index(["name", "age"])  # List of strings - NOW REJECTED
```

**After (v1.14.0):**

```python
collection.create_index([("name", 1), ("age", -1)])  # List of tuples - REQUIRED
```

Simple single-field indexes (e.g., `collection.create_index("name")`) remain unchanged.

## Overview

This release marks the "SQL-Tier 1" evolution of the engine, focusing on database-native processing:

- **SQL-Tier `$project` Stage** — Implemented native SQL processing for the `$project` aggregation stage, supporting field inclusion, exclusion, and renaming directly in the database.
- **Advanced `_id` SQL Filtering** — Migrated MongoDB operators (`$in`, `$nin`, `$ne`, and range queries) for the `_id` field to the SQL layer.
- **Dual-Column ID Support** — Intelligent SQL generation that automatically checks both the integer `id` column and the `ObjectId` string `_id` column to ensure correct filtering across mixed-type ID systems.
- **FTS on Unwound Data** — Refined Full-Text Search extraction using `json_tree` to handle complex "unwound" temporary tables more efficiently.
- **API Alignment** — Corrected the compound index API and modernized documentation/examples to match PyMongo standards.
- **Parity Verification** — Introduced `test_project_sql_vs_python.py` to ensure identical results across execution tiers.

## High Severity Optimizations & Fixes

### SQL-Tier `$project` Implementation

**Problem:** Previously, the `$project` stage was handled by the Python fallback layer, which required loading full documents into memory before filtering fields.

**Solution:** Implemented SQL-level projection in `TemporaryTableAggregationProcessor`. SQLite now handles field selection and aliasing, drastically reducing the amount of data transferred to Python.

### Advanced `_id` Operator Support in SQL

**Problem:** Queries on the `_id` field using operators like `$in` or `$gt` often caused a `ProgrammingError` or fell back to slow Python processing.

**Solution:** Added `_build_id_operator_clause` to handle complex `_id` queries. It intelligently splits search values into integer and string lists to query both potential ID columns (`id` and `_id`) in a single optimized SQL statement.

---

## Medium Severity Enhancements

### Improved FTS Extraction for Temp Tables (NeoSQLite Extension)

**Problem:** Text search on unwound data shapes was previously a "known limitation" that required slow Python fallback. In standard MongoDB, `$text` can only be used as the *first* stage in an aggregation pipeline.

**Fix:** Integrated recursive `json_tree` extraction into the FTS logic for temporary tables. This enables database-native text search even *after* `$unwind` stages by automatically building a temporary FTS5 index on all string values in the unwound documents.

**Semantic Note:** When `$text` is used after `$unwind`, it searches the *unwound* elements directly. This differs from MongoDB's collection-level indexing behavior. For standard MongoDB behavior, place `$text` as the first stage. NeoSQLite will now issue a `UserWarning` when this extension is utilized.

### Refactored ID Categorization

**Optimization:** Introduced `_categorize_ids` and `_categorize_id_value` helpers to DRY up the complex logic required for dual-column ID filtering, improving maintainability and reducing edge-case bugs in `$ne` queries.

## Migration Guide

### Upgrading from v1.13.11

1. **Update Compound Indexes:** Search your codebase for `create_index` or `create_indexes` calls that pass a list of strings for a compound index. Update them to the tuple format: `[("field", 1)]`.
2. **Review Projection Stages:** If you have extremely complex `$project` stages with custom Python logic, verify they still work as expected (SQL-tier `$project` handles inclusion, exclusion, and simple field referencing).
