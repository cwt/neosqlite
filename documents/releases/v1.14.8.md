# NeoSQLite v1.14.8 Release Notes

## TL;DR

NeoSQLite v1.14.8 is a **bug fix release** that resolves a critical `$facet` stage failure when JSONB is enabled.

**Key Fix:** `$facet` now correctly handles JSONB binary format when extracting documents from temporary tables. Without this fix, pipelines with `$unwind` + `$group` sub-pipelines would skip all documents due to UTF-8 decode errors on JSONB binary data.

- **Upgrade if you use:** `$facet` with `$unwind` and/or `$group` sub-pipelines on systems with JSONB support enabled.
- **Result Correctness:** Documents are now properly extracted from JSONB temporary tables — no more false "corrupted document" warnings.

---

## Overview

This release fixes a single but critical bug:

- **$facet JSONB Decode Error** — Fixed false `UnicodeDecodeError` when `$facet` extracts documents from temporary tables where JSONB is stored as binary BLOB instead of TEXT.

---

## High Severity Fixes

### Fixed $facet JSONB Binary Decode Error

**Bug:** `$facet` stage reported "Skipping corrupted document" warnings and skipped all documents when extracting from temporary tables created by `$unwind` + `$group` sub-pipelines with JSONB enabled.

**Error Message:**

```text
WARNING: Skipping corrupted document in $facet stage (id=68e378ff16ac64c1db650c20): 'utf-8' codec can't decode byte 0xdc in position 0: invalid continuation byte
WARNING: $facet stage skipped 8 corrupted document(s) out of 8 total
```

**Root Cause:** When JSONB is supported, temporary tables store the `data` column as JSONB BLOB (binary format), not TEXT. The byte `0xdc` at position 0 is a JSONB type indicator, not corrupted data. The `_process_facet_stage()` method was reading the raw BLOB and passing it directly to `neosqlite_json_loads()`, which expects TEXT JSON.

**Solution:** When JSONB is supported, use `json(data)` to convert JSONB BLOB to TEXT before parsing:

```python
# Before: Read raw BLOB → json.loads() fails
cursor = self.db.execute(f"SELECT _id, data FROM {current_table}")

# After: Convert JSONB to TEXT when needed
if self._jsonb_supported:
    cursor = self.db.execute(f"SELECT _id, json(data) FROM {current_table}")
else:
    cursor = self.db.execute(f"SELECT _id, data FROM {current_table}")
```

**Impact:**
- **Before:** `$facet` with `$unwind` + `$group` → all documents skipped → empty results
- **After:** Documents properly extracted → correct facet results

**Example (neo-bloggy pattern):**

```python
# This pipeline now works correctly with JSONB
pipeline = [
    {"$match": {"status": "published"}},
    {
        "$facet": {
            "posts": [{"$sort": {"datetime": -1}}],
            "related_tags": [
                {"$unwind": "$tags"},
                {"$group": {"_id": "$tags", "count": {"$sum": 1}}},
                {"$sort": {"count": -1, "_id": 1}},
            ],
        }
    },
]

results = list(collection.aggregate(pipeline))
# results[0]["posts"] - sorted published posts ✓
# results[0]["related_tags"] - tag counts ✓
```

## Migration Guide

### Upgrading from v1.14.7

This is a non-breaking bug fix release. All existing code continues to work without modification.

### For Users Experiencing $facet Empty Results

If you were seeing warnings like:

```text
WARNING: Skipping corrupted document in $facet stage (id=...): 'utf-8' codec can't decode byte 0xdc in position 0
WARNING: $facet stage skipped 8 corrupted document(s) out of 8 total
```

**After v1.14.8:** The warnings disappear and `$facet` returns correct results. Documents are properly extracted from JSONB temporary tables using `json(data)` conversion.

**No code changes needed** — this is a transparent fix that requires no application modifications.
