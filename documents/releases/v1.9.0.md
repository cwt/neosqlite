# NeoSQLite v1.9.0 Release Notes

## TL;DR

## 🔵 Major Feature & Compatibility Release — Highly Recommended Upgrade

- **Upgrade if you need:** ACID transactions, advanced aggregation stages (`$setWindowFields`, `$graphLookup`, `$fill`, streaming `$facet`), native `$jsonSchema` validation, or SQL-tier optimization for set/string operators.

## Overview

NeoSQLite v1.9.0 is a **major feature and API completeness release** that achieves PyMongo 4.x API parity for ACID transactions and introduces powerful aggregation capabilities. This version implements **dozens of new MongoDB-compatible operators and stages**, with a focus on moving complex logic from Python fallback (Tier-3) to native SQL optimization (Tier-1).

**Key Milestone**: With this release, NeoSQLite achieves **373 automated compatibility tests** (362 passed, 11 skipped), maintaining **100% compatibility** for all comparable features. The addition of ACID transactions via `ClientSession` and native `$jsonSchema` validation makes NeoSQLite a truly production-ready, drop-in replacement for PyMongo in embedded environments.

## Major Achievement: ACID Transactions via `ClientSession`

Full support for PyMongo 4.x transaction API using SQLite's native `SAVEPOINT` mechanism:

### Transaction Operations

- ✅ `start_transaction()` / `commit_transaction()` / `abort_transaction()`
- ✅ `with_transaction()` - Callback-based transactional execution
- ✅ Nested transaction support via SQLite `SAVEPOINT`

### API Integration

- Session parameter integrated across all CRUD, find, aggregation, and GridFS operations
- Write concern mapping to SQLite `PRAGMA synchronous` settings
- Proper session validation and lifecycle management

### Durability Guarantees

- **ACID Compliance**: Atomic, Consistent, Isolated, Durable transactions
- **Nested Transactions**: Multiple savepoints within a single session
- **Automatic Cleanup**: Proper rollback on abort or exception

## Aggregation Framework: Advanced Stages & Tier-1 Expansion

v1.9.0 brings a massive expansion to the aggregation framework, with a focus on moving logic from Python (Tier-3) to SQL (Tier-1):

### 1. New High-Impact Stages

**`$setWindowFields`** - Comprehensive window function support:
- Partition and sortBy specifications
- Window operators: `$rank`, `$denseRank`, `$documentNumber`, `$percentRank`, `$cumeDist`
- N-operators: `$firstN`, `$lastN`, `$minN`, `$maxN`
- Advanced operators: `$top`, `$bottom`, `$topN`, `$bottomN`
- Math operators: `$derivative`, `$integral`, `$covariancePop`, `$covarianceSamp`, `$expMovingAvg`

**`$graphLookup`** - Recursive hierarchical searches:
- Native SQL implementation using recursive CTEs (Tier-1)
- Temporary table support for complex pipelines (Tier-2)
- Python fallback for maximum compatibility (Tier-3)
- Proper `_id` mapping and column resolution

**`$fill`** - Intelligent data filling for sequential datasets:
- `locf` (last observation carried forward) method
- Constant `value` filling
- Native SQL support using window functions
- Linear interpolation fallback in Python tier

**Streaming `$facet`** - Parallel sub-pipeline processing:
- Batch processing with memory-efficient `fetchmany(101)`
- Independent Tier-1/2/3 optimization per sub-pipeline
- Automatic temp table cleanup
- MongoDB-compatible `batchSize` parameter (default 101)

### 2. Tier-1 (SQL) Optimization Expansion

Many operators that previously relied on Python fallback now run directly in SQL:

**Set Operators** (All 7 operators):
- `$setEquals`, `$setIntersection`, `$setUnion`
- `$setDifference`, `$setIsSubset`
- `$anyElementTrue`, `$allElementsTrue`
- Uses `json_each()` with `EXISTS`/`NOT EXISTS` patterns

**String Operators**:
- `$split` - Implemented via recursive CTE
- `$replaceAll`, `$replaceOne` - Native SQL string functions

**Variable Scoping**:
- `$let` - SQL-tier optimized via variable inlining
- Proper `$$ROOT` and `$$CURRENT` resolution

**Accumulators**:
- `$addToSet` - Uses `json_group_array(DISTINCT ...)`
- `$stdDevPop`, `$stdDevSamp` - SQL math formulas

**Other Optimizations**:
- `$unwind` - Tier-2 support with `preserveNullAndEmptyArrays` and `includeArrayIndex`
- `$group` - Expression keys support in Tier-2
- `$first`, `$last` - Proper handling with `$sort` detection

## Security: Native `$jsonSchema` Validation

v1.9.0 introduces robust JSON Schema support for data integrity:

### Query Validation

- Use `$jsonSchema` within `find()` or `$match` stages
- Memory-efficient generator filter on `Cursor`
- Tier-2 and Tier-3 engine integration

### Write Validation

- **Schema Compiler**: Translates `$jsonSchema` into native **SQLite CHECK constraints**
- **Database-Level Enforcement**: Automatic rejection of invalid writes via `IntegrityError`
- **Supported Constraints**: `required`, `properties`, `bsonType`, numeric constraints
- **High Performance**: No Python-level triggers for common validation rules

### Supported JSON Schema Keywords

- `required` - Required field validation
- `properties` - Field-specific schema definitions
- `bsonType` - MongoDB type aliases (e.g., `objectId`, `date`)
- Numeric constraints: `minimum`, `maximum`, `exclusiveMinimum`, `exclusiveMaximum`

## Known Limitations

### Architectural Differences

**`$first` / `$last` in `$group`**: When preceded by `$sort`, requires Tier-3 Python fallback due to SQLite's inability to guarantee order in grouped aggregations without explicit subqueries.

**`$where` Operator**: NeoSQLite uses Python functions instead of MongoDB's JavaScript execution. Proactive error handling redirects users to `$expr` for better performance.

**`watch()` Change Streams**: Requires MongoDB replica set. NeoSQLite provides independent testing via SQLite triggers.

### Performance Considerations

**`$bsonSize`**: Uses serialized JSON length for consistency between tiers, which may differ slightly from MongoDB's binary BSON size calculation.

**`$binarySize`**: Base64-aware heuristic approximation for SQL tier; exact calculation in Python tier.

## Migration Guide

### Upgrading from v1.8.0

**Steps**:
1. Update NeoSQLite: `pip install neosqlite==1.9.0`
2. Run tests to confirm compatibility (all should pass)
3. Explore new transaction APIs for atomic operations
4. Consider `$jsonSchema` for data integrity requirements

**Effort**: Minimal - typically 5-15 minutes for most users.

### Adopting New Features

**Transactions**:

```python
with client.start_session() as session:
    with session.start_transaction():
        collection.insert_one({"balance": 100}, session=session)
        collection.update_one({"_id": 1}, {"$inc": {"balance": 50}}, session=session)
```

**Window Functions**:

```python
pipeline = [
    {"$setWindowFields": {
        "partitionBy": "$category",
        "sortBy": {"date": 1},
        "output": {
            "rank": {"$rank": {}},
            "runningTotal": {"$sum": "$amount", "window": {"documents": ["unbounded", "current"]}}
        }
    }}
]
```

**JSON Schema Validation**:

```python
db.create_collection("users", validator={
    "$jsonSchema": {
        "required": ["name", "email"],
        "properties": {
            "name": {"bsonType": "string"},
            "email": {"bsonType": "string", "pattern": "^.+@.+\\..+$"}
        }
    }
})
```
